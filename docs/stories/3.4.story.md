## Contexto / Problema

Hoje `API_ENDPOINTS.processarContratacao` (`src/lib/api.ts:4`) aponta para uma function **que não
existe no repo** — é a ponte deployed-only para o n8n. O `SignupForm` envia os dados e faz **polling
frágil** por `get-contratacao-status` usando um `email_timestamp` no `localStorage`
(`SignupForm.tsx:151`). Esta story cria a function in-house que faz o Fluxo 1 do n8n
(salvar contratação → preparar dados → criar contrato ZapSign → salvar token/URL) e **retorna o
`contratacao_id` + `cliente_plano_id` + `signing_url` reais** ao browser, eliminando o polling.

## Acceptance Criteria

1. **Nova Edge Function `supabase/functions/processar-contratacao/index.ts`** (Deno), padrão das
   demais (CORS, `serve`, service-role client). Registrada em `supabase/config.toml` com
   `verify_jwt = false` (chamada pelo funil público, como era o webhook n8n).
2. **Entrada:** payload do formulário (dados pessoais/endereço + `tipo_pessoa` `'fisica'|'juridica'`
   + `plano_id`). **Validação:** campos obrigatórios + `plano_id` existente; erro 400 claro se faltar.
3. **Upsert da contratação:** insere em `contratacoes_clientes` (id gerado pelo DB) com os campos do
   form + `status_contratacao = 'CONTRATO_ENVIADO'`; reusa contratação existente por e-mail se já
   houver uma em andamento (evita duplicar pessoa).
4. **Cria a assinatura pendente** em `cliente_planos`: `status = 'aguardando_assinatura'`,
   `preco_snapshot_centavos = planos.preco_em_centavos`, `produto_id` do plano, `proximo_vencimento`
   deixado a cargo do trigger (`trg_cliente_planos_set_vencimento`, provisório até o pagamento).
5. **Contrato ZapSign por assinatura** (decisão #8): `POST https://api.zapsign.com.br/api/v1/models/
   create-doc/` com `template_id = planos.zapsign_template_id_pf|pj` (conforme `tipo_pessoa`),
   `external_id = cliente_plano_id`, signatário (nome/email), `send_automatic_email: true`,
   `redirect_link = https://clientes.onofficebelem.com.br/processando-pagamento?id={contratacao_id}`,
   `data = dataVars` (mapa `{{NOME_RESPONSAVEL}}`… do reference doc, com campos PJ só se `juridica`).
   Token via secret `ZAPSIGN_API_TOKEN` (nunca hardcoded).
6. **Persiste o contrato na assinatura:** grava em `cliente_planos` `zapsign_document_token`,
   `zapsign_signing_url` (= `signers[0].sign_url`), `zapsign_template_id`.
7. **Resposta síncrona:** `200 { success, contratacao_id, cliente_plano_id, signing_url }` — fim do
   polling por `localStorage`. Em erro: `4xx/5xx` com `{ success:false, error }` (sem vazar segredo).
8. **Idempotência básica:** se a mesma contratação+plano já tem assinatura `aguardando_assinatura`
   com contrato, retorna a existente em vez de criar outro contrato (evita duplicar no retry).
9. **Migração de config:** popular `planos.zapsign_template_id_pf/pj` (hoje possivelmente nulos) a
   partir dos IDs conhecidos do n8n, mapeados por `periodicidade` (mensal/anual/bianual) — para a AC5
   funcionar com dados reais.
10. **Gates:** `deno check` na function; `npx tsc --noEmit`/`build` do app inalterados (function não
    entra no bundle do Vite). **Deploy + secret + teste com ZapSign** ficam para um passo gated
    (cria contrato real) — documentado, não executado às cegas.

## Out of Scope

- Rewire do `SignupForm`/páginas de acompanhamento ao novo fluxo → **Story 3.7**.
- Webhook ZapSign (assinado → cobrança) → **Story 3.5**. Link InfinitePay → **Story 3.6**.
- Multi-produto/carrinho → Fase 2. Esta function cria **1 assinatura por chamada**.

## Dependências

- 3.1 (colunas `zapsign_*` em `cliente_planos`), 3.2 (trigger de vencimento), 3.3 (backfill).
- `docs/architecture/n8n-sistema-on-reference.md` (template IDs, dataVars, redirect URL).
- **Secret `ZAPSIGN_API_TOKEN`** no Supabase (fornecido pelo proprietário) para deploy/teste.

## Riscos

- **Template IDs ausentes** em `planos` → AC9 (backfill) mitiga; function valida e erra claro.
- **Webhook idempotente** (próximas stories) — aqui já gravamos `external_id = cliente_plano_id` para
  casar o retorno do ZapSign à assinatura certa.
- **Modo-sombra:** enquanto 3.7 não faz cutover, esta function coexiste com o n8n (sem conflito —
  caminhos separados).
