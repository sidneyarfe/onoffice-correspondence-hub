## Contexto
InfinitePay Checkout API (`POST https://api.checkout.infinitepay.io/links`) cria link one-off
(cartão + PIX, até 12x), sem recorrência nativa → confirma o modelo "nós geramos cada cobrança".

## Acceptance Criteria
1. **Helper `supabase/functions/_shared/infinitepay.ts`** `criarLinkPagamento({ handle, items[], orderNsu,
   webhookUrl, redirectUrl })` → `POST /links` com preços em **centavos**; retorna
   `{ slug, order_nsu, payment_url }`. Segredos `INFINITEPAY_HANDLE` + `INFINITEPAY_API_TOKEN`.
   (Nomes exatos de campos da API validados no deploy contra a doc oficial.)
2. **Edge Function `infinitepay-webhook/index.ts`** (`verify_jwt=false`): recebe o webhook de pagamento.
   **Idempotência por `order_nsu`/`slug`:** se o `pagamentos` já está `pago`, retorna 200 sem reprocessar.
3. Ao **aprovado**: atualiza `pagamentos` (`status='pago'`, `data_pagamento`, valores reais); atualiza
   `cliente_planos` (`status='ativo'`, `paid_at`, `metodo_pagamento`, `data_ultimo_pagamento`,
   `proximo_vencimento` recalculado via `calcular_vencimento_por_periodicidade`).
4. **Provisiona o usuário** reusando a Edge Function `create-user-from-contratacao` (senha temporária)
   — gatilho passa a ser a **assinatura paga**, não a contratação. Espelha `status_contratacao='ATIVO'`.
5. **Conciliação:** `POST /payment_check` disponível no helper para reconferência (defensivo).
6. Gates: padrão Deno; deploy/secret/e2e **gated** (cobrança real).

## Out of Scope
- Renovação recorrente automática (job) → **3.13**. Inadimplência/carência → **3.14**.
- E-mails (pagamento aprovado / boas-vindas) → follow-up peri­férico (#2); credenciais já vão no
  retorno do provisionamento.
