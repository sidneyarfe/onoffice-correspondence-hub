# Story 5.9 — Alinhamento de CRM (iniciar contratação) e Correspondências ao novo modelo

**Epic:** [005](./epic-005-modelo-dados-ideal-clientes.md) · **Fase:** Onda D
**Status:** Done (gates ✅)
**Dono:** @dev (Dex) + @ux-design-expert (Uma) · Orquestração: @aiox-master (Orion)

## Story
Como **admin/comercial**, quero que o CRM e o módulo de correspondências reflitam o **novo modelo de
clientes e produtos** (assinaturas/avulsos, endereço próprio × fiscal, design dark), para iniciar
contratações alinhadas e gerir correspondências com a UX atualizada.

## Acceptance Criteria
1. **"Iniciar contratação"** rebuildado como **carrinho multi-produto** (decisão do usuário): seleção de N assinaturas + N avulsos (busca + abas), em estilo dark. ✅
2. O formulário coleta o **endereço próprio do cliente** (o endereço fiscal é recurso atribuído à parte — Epic 005). ✅
3. Ao confirmar: cria/atualiza o cliente (upsert por e-mail) + **N assinaturas (`aguardando_assinatura`)** + um **pedido com itens avulsos**, tudo client-side (RLS admin); liga o negócio do CRM e move para "Contrato Enviado". Contrato/cobrança ficam para a ficha. ✅
4. **`ClientSearchSelect`** redesenhado para o tema dark (avatar + nome/e-mail + **status pill** + plano), com busca e estado vazio. ✅
5. **Reenviar notificação da correspondência** no `CorrespondenceDetailModal` (plataforma + e-mail + log). ✅

## Dev Notes
- `processar-contratacao` (legado) já cria uma `assinatura` em `aguardando_assinatura`; o carrinho replica o padrão client-side para múltiplos itens, evitando reescrever a edge fn.
- Mudança de comportamento: o ZapSign **não** é mais disparado neste passo — o contrato é enviado por assinatura na ficha (novo fluxo). Registrado como decisão a confirmar.
- `contratacoes_clientes` aceita insert do admin (mesma policy do "Adicionar cliente").
