# Story 5.7 — Ficha do Cliente reformulada: Financeiro, 3 status, Resumo, Documentos & Atividades

**Epic:** [005](./epic-005-modelo-dados-ideal-clientes.md) · **Fase:** Onda D
**Status:** Done (gates `tsc`/`lint`/`build` ✅ · migrações de RLS aplicadas em prod — ver 5.8)
**Dono:** @ux-design-expert (Uma) + @dev (Dex) · Orquestração: @aiox-master (Orion)

## Story
Como **admin**, quero a ficha do cliente reorganizada para mostrar **claramente a relação do cliente
conosco**, respeitando assinaturas × avulsos, com três dimensões de status alinhadas e abas de
Documentos e Atividades reais — para gerir o cliente sem ambiguidade.

## Acceptance Criteria
1. Aba renomeada **"Plano & Financeiro" → "Financeiro"**; cabeçalho com **apenas contato** (WhatsApp + E-mail); "Gerar cobrança" removido; "Excluir" movido para a aba **Cadastro** (zona de perigo). ✅
2. **Três status independentes e alinhados** (decisão do usuário): funil (CRM) · **cliente** (ativo/inadimplente/suspenso/cancelado, derivado de assinaturas+faturas) · **assinatura** (aguardando_assinatura/aguardando_pagamento/vigente/em_atraso/suspensa/cancelada). ✅
3. **"Resumo do contrato" → "Resumo do cliente"**: nome, telefone+WhatsApp, e-mail, endereço, data de cadastro, produtos contratados (nome completo, quebra de linha), **faturamento gerado (estimado)** e status atual. ✅
4. Ações sensíveis da assinatura (**Renovar/Suspender/Cancelar**) em pop-ups próprios, **gatilho por senha do admin** (reauth). ✅
5. **Documentos**: anexar documentos do cliente (upload → `documentos_cliente`, bucket `documentos_fiscais`). ✅
6. **Atividades**: registrar ações do cliente **e** ações do admin (helper `registrarAtividade` → RPC `registrar_atividade` SECURITY DEFINER); humanização do `acao`. ✅
7. **Correspondências**: registrar correspondência recebida a partir da ficha (modal travado no cliente). ✅
8. **Endereço Fiscal**: painel para gerenciar os documentos de endereço fiscal disponíveis ao cliente (`documentos_disponibilidade`), com `ClientDocuments` honrando a disponibilidade. ✅

## Dev Notes
- Status do cliente **derivado na leitura** (sem migração nova): `deriveClienteLifecycle` em `clienteStatus.tsx`.

- `EnderecoFiscalDocs` detecta a assinatura de endereço fiscal por heurística de nome.
