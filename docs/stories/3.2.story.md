## Contexto / Problema

O cálculo de `proximo_vencimento` por `periodicidade` está **triplicado em JS** —
`useClientPlanos.ts:96-127`, `useAdminClients.ts:307-322` e o legado string-based
`calcularProximoVencimento` (`useAdminClients.ts:413`) — espelhando **duas RPCs SQL concorrentes**:
`calcular_vencimento_por_periodicidade(data_inicio, periodicidade)` (membership-aware, **canônica**)
e `calcular_proximo_vencimento(data_contratacao, plano)` (legado, baseado em strings `'1 ANO'`).
Divergências entre essas cópias produzem vencimentos inconsistentes — risco direto para o motor de
cobrança (Fase 3).

## Acceptance Criteria

1. **Trigger de rede de segurança em `cliente_planos`** (migração): `BEFORE INSERT`, se
   `NEW.proximo_vencimento IS NULL`, preenche via
   `calcular_vencimento_por_periodicidade(COALESCE(NEW.data_inicio, CURRENT_DATE), <periodicidade do
   plano>)`. Garante vencimento correto para inserts que **não** vêm do JS (pipeline da Fase 1).
2. **`calcular_vencimento_por_periodicidade` é a única RPC canônica.** O legado
   `calcular_proximo_vencimento` recebe `COMMENT` de depreciação (não removido — pode ter
   referências; remoção fica para a story de limpeza da Fase 3).
3. **Duplicação JS eliminada:** criar **um** helper `src/utils/vencimento.ts`
   `calcularProximoVencimento(dataInicio: Date, periodicidade: string): Date` com a **mesma**
   semântica da RPC (semanal/mensal/trimestral/semestral/anual/bianual; fallback anual). Substituir
   as 3 cópias por chamadas a ele (mantendo o override explícito do admin quando informado).
4. **Paridade JS↔SQL:** o helper e a RPC produzem a mesma data para as 6 periodicidades (documentar
   no helper que o **DB é a autoridade**; o JS é espelho para UX otimista).
5. **Gates:** `npx tsc --noEmit`, `eslint` (arquivos tocados), `npm run build` verdes; migração
   aplicada via `supabase db push` (histórico já reconciliado na 3.1).
