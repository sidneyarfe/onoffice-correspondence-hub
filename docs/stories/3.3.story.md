**Fase:** 0 — Fundação de dados

> reversibilidade** antes de qualquer escrita.

## Contexto / Problema

`cliente_planos` está **dormente**: o único escritor é a edição manual no admin. Os clientes ATIVOS
existentes têm plano/vencimento apenas em `contratacoes_clientes` (`plano_id`/`plano_selecionado`/
`proximo_vencimento`). Sem uma assinatura por cliente, o motor de receita (MRR, radar de vencimento,
renovação) não enxerga ninguém. Esta story cria **1 `cliente_planos` por cliente ativo** a partir da
contratação, de forma **idempotente** e **validada**.

## Acceptance Criteria

1. **Dry-run de auditoria (read-only, OBRIGATÓRIO antes de escrever):** migração/consulta que
   reporta, sem mutar nada (rollback via `RAISE EXCEPTION`): (a) nº de contratações ATIVAS elegíveis,
   (b) quantas já têm `cliente_planos` correspondente, (c) quantas seriam criadas, (d) quantas
   elegíveis têm `plano_id` nulo (não-backfilláveis sem heurística), (e) amostra de 5 linhas que
   seriam criadas (cliente, plano, vencimento derivado). **Checkpoint humano: revisar antes do passo 2.**
2. **Backfill idempotente:** `INSERT INTO cliente_planos (...) SELECT ... FROM contratacoes_clientes c
   WHERE c.status_contratacao = 'ATIVO' AND c.plano_id IS NOT NULL AND NOT EXISTS (SELECT 1 FROM
   cliente_planos cp WHERE cp.cliente_id = c.id AND cp.plano_id = c.plano_id)`. Reexecutável sem
   duplicar. Deriva: `status='ativo'`, `data_inicio = COALESCE(c.created_at::date, CURRENT_DATE)`,
   `proximo_vencimento = COALESCE(c.proximo_vencimento, calcular_vencimento_por_periodicidade(...))`,
   `preco_snapshot_centavos` a partir de `planos.preco_em_centavos`, `produto_id` do plano,
   `valor_pago_centavos`/`data_ultimo_pagamento` quando disponíveis.
3. **Reversibilidade:** as linhas criadas pelo backfill são identificáveis (ex.: `created_at` do batch
   ou coluna/observação) para revert cirúrgico se a validação pós-backfill acusar erro.
4. **Validação pós-backfill:** nº de assinaturas ATIVAS == nº de contratações ATIVAS com `plano_id`;
   nenhuma com `proximo_vencimento` nulo; amostra conferida.
5. **Campos single-plan marcados como legados** (documentação/`COMMENT`) — leitura continua durante a
   transição; remoção é story de limpeza (Fase 3).
6. **Gates:** migração aplicada via `db push`; sem alteração em `contratacoes_clientes` (só leitura).

## Out of Scope

- Backfill de clientes NÃO-ativos (pendentes/cancelados) — só ATIVOS nesta story.
- Clientes ATIVOS com `plano_id` nulo (resolver por heurística `plano_selecionado` → `plano_id` é
  follow-up se o dry-run mostrar volume relevante).
- Repointar `pagamentos` históricos para `cliente_plano_id` (story de limpeza).

## Dependências

- 3.1 (colunas) + 3.2 (cálculo de vencimento canônico). Habilita Fase 2/3 inteira.

## Riscos

- **Vencimento/plano errado em cliente vivo:** mitigado pelo dry-run + idempotência + reversibilidade.
- **`plano_id` nulo** em ATIVOS antigos (só tinham `plano_selecionado`): contados no dry-run; fora de
  escopo o mapeamento por heurística.
