# Story 3.7 — Cutover do funil ao pipeline in-house (remover n8n do caminho crítico)

**Epic:** 003 — Motor de Receita Recorrente · **Fase:** 1 — Pipeline in-house
**Tipo:** Frontend · **Status:** Draft

> **Decisão de UX (2026-06-24):** onboarding **embedded** — contrato assinado via **Widget ZapSign
> (iframe)** dentro da nossa tela; pagamento abre o checkout InfinitePay em **popup controlado**
> (InfinitePay não embute — ver épico, Decisões resolvidas 2026-06-24). Cutover atrás de
> `VITE_USE_INHOUSE_PIPELINE`, com o n8n **quente** para rollback.

## Contexto
Hoje `SignupForm.handleSubmit` envia ao n8n (`useContratacao.ts:57`) e faz polling frágil por
`email_timestamp` no `localStorage` (`SignupForm.tsx:151`). As Stories 3.4–3.6 trouxeram o pipeline
in-house (`processar-contratacao`, `zapsign-webhook`, `infinitepay-webhook`). Esta story religa as
páginas de acompanhamento ao novo fluxo e tira o n8n do caminho crítico.

## Pré-requisito (gate — bloqueante)
Funções 3.4/3.5/3.6 **deployadas + secrets + e2e em staging OK**. **Não implementar antes disso**
(violaria a estratégia "modo-sombra + n8n quente" do épico).

## Acceptance Criteria
1. Feature flag `VITE_USE_INHOUSE_PIPELINE` (default **off**). Off → fluxo n8n atual intacto (rollback).
2. On → `SignupForm.handleSubmit` chama `processar-contratacao` com **`plano_id` (UUID)** + dados
   pessoais/endereço; recebe `contratacao_id`, `cliente_plano_id` e o **token do signatário**. Elimina
   o polling por `email_timestamp` (`SignupForm.tsx:151`).
3. **Assinatura embedded:** a tela renderiza o **Widget ZapSign** num iframe
   (`https://app.zapsign.com.br/verificar/{signer_token}`, `allow="camera"`) — **sem redirecionar**.
   Escuta o evento `zs-doc-signed` (postMessage) para avançar a UI **na hora**, sem esperar o webhook.
4. **Pagamento em popup:** após assinado, a tela faz polling da assinatura até `payment_link` (criado
   pelo `zapsign-webhook`) e exibe um botão **"Pagar"** que abre o checkout InfinitePay via `window.open`
   (popup) — o clique evita bloqueio de popup. A tela segue fazendo polling de `cliente_planos.status`.
5. **Conclusão:** quando o `infinitepay-webhook` marca `status='ativo'`, a tela detecta (polling) e
   mostra sucesso + credenciais (sem `localStorage` frágil).
6. **Ajuste backend (em `processar-contratacao` da 3.4):** persistir/retornar `signers[0].token` (token
   do *signatário*, necessário ao widget — hoje só guardamos o token do *documento*).
7. Sem regressão para onboarding **"em voo"** — plano de drenagem (ver épico, Estratégia de migração).
8. **`ui-ux-pro-max` rodado antes de tocar UI**; gates `lint` + `tsc --noEmit` + `build`; e2e em staging com flag on.
