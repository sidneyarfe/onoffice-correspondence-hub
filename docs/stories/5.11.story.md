# Story 5.11 — Reformulação das abas Equipe e Financeiro (dados reais Epic 005)

**Epic:** [005](./epic-005-modelo-dados-ideal-clientes.md) · **Fase:** Onda E
**Status:** Done (gate CONCERNS ✅)
**Dono:** @dev (Dex) + @ux-design-expert (Uma) · Orquestração: @aiox-master (Orion)

## Story
Como **admin**, quero que as abas **Equipe** e **Financeiro** reflitam o **design atualizado** (dark,
tabela, brand tokens) e operem sobre os **dados reais do modelo Epic 005** (assinaturas/faturas/pedidos),
com gestão de cobranças e visão de inadimplência corretas — para administrar o sistema sem telas
simuladas nem fluxos quebrados.

## Acceptance Criteria
1. **Equipe** reformulada para **layout de tabela** (padrão das páginas atualizadas): header + stats + busca + tabela com avatar/nome/e-mail, pílula "Administrador", data e ações inline (Editar/Excluir). Botão **"Sincronizar Admin" removido** e `SyncAdminModal` **deletado**. Realtime em `profiles`; auto-exclusão do próprio admin bloqueada; pílula "você". ✅
2. **Financeiro** reescrito sobre **dados reais** (view `faturas` sobre `pagamentos`, `assinaturas`, `pedidos`/`pedido_itens`) no lugar da edge fn simulada `get-financial-overview`: KPIs (Receita recebida, **MRR** normalizado por periodicidade, Em aberto, Inadimplência), Receita por produto, Inadimplentes (com notificação), tabela de faturas (busca/ordenação/filtro de status), filtro de **período** e **export CSV** (BOM p/ Excel). ✅
3. A gestão de cada fatura **reutiliza `FaturaModal`** da ficha; o modal exibe **explicitamente** **Cliente relacionado**, **Produto cobrado** e **Período cobrado** (props novas `cliente` + `FaturaGerencia.produtoNome`). ✅
4. **Inadimplência assinatura-cêntrica:** toda assinatura viva (não cancelada/suspensa) **vencida** — `proximo_vencimento` no passado **OU** com fatura `vencida` — entra no KPI; valor = soma das faturas vencidas, ou o preço de 1 período se vencida só por data (sem fatura); + faturas avulsas vencidas. Não é mais apenas Σ faturas vencidas. ✅
5. **Equipe — histórico de atividades dentro do modal:** deixou de ser botão separado e passou para o **modal de edição reformulado** (abas **Dados / Senha / Atividades**, via `useAdminActivities`), com estados de loading/vazio/erro. `AdminActivityModal` **deletado**. ✅
6. **Todos os números** das abas e modais usam **DM Mono** (`on-num` = `font-dm tabular-nums`). ✅

## Dev Notes
- Fonte da verdade financeira = view **`faturas`** (`security_invoker=true` → herda RLS de `pagamentos`; admin lê tudo via `20260627210000`). MRR = Σ assinaturas `ativo` com preço normalizado por meses/periodicidade.
- Inadimplência migrou do nível **fatura** para o nível **assinatura**, alinhando com `clienteStatus.assinaturaEmAtraso` / `deriveClienteLifecycle` (mesma semântica usada no Kanban/lista de clientes). Sem dupla contagem (fatura vencida tem prioridade sobre preço de período).
- `FaturaModal` reutilizado no Financeiro com `assinatura=null` (pula update de status de assinatura); período derivado de `criadaEm + periodicidade` (`calcularProximoVencimento`).
- Atividade de admin: `useAdminActivities` lê `atividades_cliente` por `user_id = adminId`; depende da RLS `20260628120000`. **Caveat:** ações de admin são logadas no `user_id` do CLIENTE (sufixo "(admin)"), então a aba tende a ficar **vazia para admin puro** — registrado como débito (logging atribuído ao admin).
- `get-financial-overview` ficou **órfã** (sem consumidores); mantida deployada (inofensiva) para não mexer em config/deploy.
